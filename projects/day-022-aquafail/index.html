day 022 — first total flop. watercolor clouds that aren't.

(yes — this page is styled like watercolor. static CSS on a fixed layout: that's the easy case. the hard case — matching a specific hand-painted reference, animated, in real time — is exactly what the three sessions below couldn't do.)

The shared thread across all three sessions: procedural watercolor clouds in the browser. The framing around that thread varied — different reference material each time, different scope (only attempts 1 and 2 knew about the downstream cherry-blossom animation), different question shape (attempt 3 was started with a single reference and "can you generate something like this?", framed tighter precisely because attempts 1 and 2 had already failed). None of the three produced a usable result. What follows: a meta-analysis that reads the three post-mortems side-by-side, then the post-mortems themselves.

Meta-analysis · what all three have in common

Meta · pattern across three sessions

Three agents. Three different briefs. Same failure shape.

The three sessions were not variations of a single prompt. They differed in what they were shown and what they were asked:

  • Session 1 — many real watercolor reference images, open-ended exploration of procedural watercolor in a shader; cherry-blossom downstream context mentioned.
  • Session 2 — Three.js fluid-sim angle, with the cherry-blossom context as part of the brief.
  • Session 3 — a single reference image and the direct question "can you generate something like this?", framed tighter precisely because sessions 1 and 2 had already failed; no cherry-blossom context passed through.

Three different briefings. Three different starting materials. Three agents that did not know about each other. The same six failure modes show up in every one. That's the observation that matters: it isn't a shared-briefing artifact, and it isn't a single-run personality quirk — it's what the system does by default when the goal is "match this visual reference" and the method is "procedural shader."

1.Capability claim without rendering

All three sessions promised output quality before producing any. PM1 reported optimistic match percentages (95 % → later corrected down to 70 %) with no reference-comparison harness in place. PM2 described every iteration as "clearly more watercolor-like" based on technical progress, not on visual distance to target. PM3 spent three full iterations claiming it would work without rendering a single image — actual comparison only began after an explicit "just render something" from my side.

The root is the same across all three: a procedural shader can be written, and that fact gets confused with the output looking right. Writing a shader that does pow(color, density) is easy. Producing a result that reads as watercolor is a different question, and it never got asked up front.

2.Method locked in before feasibility was checked

All three bound the problem to a specific method class early — fragment shader (PM1), single-pass noise/SDF/stroke shader (PM2 v1–v12), FBM-then-metaballs (PM3) — and then tuned within that class for hours. The literature on the problem (Curtis 1997, Computer-Generated Watercolor) is unambiguous: reproducing a specific watercolor reference needs a multi-layer fluid simulation with pigment kinetics, running over minutes, not a single-pass shader running per-pixel. Every session mentions this eventually — usually late, usually after the sunk cost is already large.

PM1 named reaction-diffusion as "probably the only path past 80 %" but dismissed it as "too complex." PM2 reached for a proper fluid sim only at v13, too late and in a stripped-down form. PM3 cites Curtis correctly — in the "what's technically true" section, only after the original goal had already been written off.

3.Parametric drift instead of method change

The observable failure mode across all three is iteration without learning. Each tweak fixes one symptom and introduces another. PM1 names this explicitly: "rim-lines → cell membrane, granulation → foxing, hard edges → Lego bars, soft edges → Gaussian clouds." PM2 spent ten iterations inside the same noise-threshold regime before switching to SDFs. PM3 switched once (FBM → metaballs) and then stayed there once metaballs stopped being obviously wrong.

The missing discipline: after 2–3 iterations with the same core method and unclear progress, change method, not parameters.

4.No objective progress metric

None of the three sessions set up a reference-comparison harness — not even the simplest one: "render the current output next to the reference, same size, same frame, every iteration." Progress was judged subjectively, turn by turn, with no calibration against the actual target. PM1 names this directly: "without a pixel-based similarity metric or clear acceptance criteria, every iteration is just 'felt better/worse.'"

The asymmetry is telling: none of the three sessions struggled with computing Laplace stability bounds, Beer-Lambert falloff, or stroke-glazing integrals — but none built the equivalent quantitative check for whether the output was actually getting closer to the goal.

5.Scope question asked late, or not at all

PM1 catches this in retrospect: hours were spent building a hero-quality watercolor when the downstream use was "backdrop behind cherry blossoms" — the scoping question arrived late. PM2 raises it in its takeaways too: "stop at v3 or v9, say honestly 'this is what we get with this technique,' and move to the next project layer." PM3 had no downstream-use signal at all (the cherry-blossom context didn't reach it), which in theory should have narrowed things — but the fidelity-vs.-effort question never came up there either. Across all three: no early, explicit what fidelity do you actually need, and at what effort budget? before committing to a method.

6.Communication drift toward optimism

All three post-mortems name overselling as a failure mode. Every iteration got reported as "closer," "clearly watercolor-like," "first version that actually reads as watercolor" — phrases that described technical progress (a known bug was fixed) but not goal progress (the output still didn't look like the reference). The deflating came from outside the session, every time.

What would have worked (consistent across all three)

An honest initial framing — "a procedural single-pass approach gets you a 60–70 % watercolor approximation in a day; matching a specific hand-painted study needs a multi-pass fluid pipeline (several days of tuning) or a pre-rendered asset" — plus a reference-comparison step after every iteration, plus a method-change rule after three unclear iterations. Any one of the three in isolation would probably have cut the combined session time in half. All three together would have surfaced the real answer in the first session: for this specific goal at this session budget, prerender or AI-generate the backdrop texture; let the shader handle what shaders are actually good at — compositing, reveal animation, the cherry-blossom layer on top.

The more uncomfortable observation

Session 3 was specifically framed as "can you generate something like this?" — tighter, narrower, following two known failures of the broader question. You'd expect a tightened brief after two failures to produce either a clearer "no" or a more honest scoping answer. It produced the same failure shape as the first two. Different starting points, different questions, different reference sets — same trap. That's a structural bias, not a briefing artifact: when asked "can you do X?" for a visual-quality goal, the default answer is yes, and start building, rather than here's the feasibility envelope, where in it do you want to land? Filing this as status: fail is the honest outcome; what it costs in "shippable project" it returns in diagnostic value.

Attempt 1 · WebGL shader with Beer-Lambert layering 3 selected frames (watercolor.html)

First run. Fragment-shader approach with pigment-correct layering physics (pow(color, density)), real Canvas-2D paper noise texture, elliptical region mask, three cloud layers. The individual pieces were technically sound; the combined output never converged on a watercolor look. Full source lives externally as watercolor.html; below are three frames from the session — the closest-to-goal (cloud_band) and two named failure modes called out directly in the post-mortem (cauliflower_rim → "cell membrane," lego_bars → "hard edges → Lego bars").

Post-mortem 1 · watercolor shader project

Starting point

Goal: a procedural watercolor backdrop in WebGL for a cherry-blossom animation. Iterative, feedback-driven work over several hours.

What worked

  • Beer-Lambert layering (pow(color, density)): technically correct additive pigment behaviour. Overlaps of the same colour deepen the hue instead of collapsing to a muddy grey average. The one real breakthrough that actually modelled watercolor physics.
  • Wet/dry gradient per blob — one wet-rich zone and one dry zone per shape — added more character than the eight iterations before it.
  • Real Canvas-2D noise as paper base: more authentic than procedural fbm.
  • Cauliflower edges via texture displacement: produced organic silhouettes.

What didn't work

  • Convergence: every iteration fixed one symptom and introduced another. Rim lines → cell membrane. Granulation → foxing. Hard edges → Lego bars. Soft edges → Gaussian clouds. Never landed in a stable equilibrium.
  • Calibration: match percentages given during the session didn't hold up (95 % → later corrected to 70 %). Subjective assessment without an objective metric eroded trust.
  • Scope misread: hours spent building a hero-quality watercolor when only a backdrop for an animation was actually needed. The scope question was asked far too late.
  • References not read critically enough: some of the watercolors used as targets were AI-generated (highly idealised, "too perfect"). The session kept chasing them instead of flagging early that such references are an unrealistic benchmark.
  • Oscillation instead of accumulation: the approach moved from A → B → C and back to A (blobs → strokes → cloud-fields → horizontal washes → cloud shapes → Gaussians). Features from earlier iterations were lost rather than carried forward.

Structural causes

  • Procedural shaders meet real watercolor: the core problem. Real watercolor arises from fluid dynamics between pigment, water, and paper fibres — a simulated physical system. A fragment shader computes per pixel, independently, without neighbourhood interaction or time integration. The characteristic watercolor features (bleeds, blooms, backruns) are emergent from physics, not synthesisable from noise fields. Anything procedurally achievable along this path is imitation of the appearance, not reproduction of the process — and imitation tips into the artificial fast.
  • Reaction-diffusion simulation (multi-pass framebuffers, temporal iteration) would conceptually have been the right approach. It was raised early as an option but not pursued — deemed "too complex" at the time. In hindsight, probably the only path that could realistically have passed 80 %.
  • No objective progress marker: without a pixel-based similarity metric or clear acceptance criteria, every iteration is just "felt better/worse" — which leads to endless refinement without closure.

What should have been done differently

  1. After three iterations with unclear results: stop and rethink structurally, not keep tuning parametrically.
  2. Early scoping question: "what exactly is this asset for?" — before the first line of code. Would have prevented 80 % of the wrong turns.
  3. Recognise AI-generated references as such and flag them early: unreachable benchmark.
  4. Put reaction-diffusion or a pre-rendered texture on the table as primary options, rather than spending hours tuning procedurally.
  5. Honest match assessment from the start — no optimistic percentages.

Recommendation for the current use case

A static, hand-painted (or single-shot AI-generated) watercolor background as a PNG. Let the shader handle only the branches, the blossoms, the wind effects — it's well suited to those. The watercolor quality comes from real painted source material, not from noise.

Outcome

A functional shader (watercolor.html) with Beer-Lambert layering, a Canvas-2D noise texture, an elliptical region mask, and three cloud layers. As watercolor imitation: mediocre. As a quiet atmospheric soft gradient: usable. As a foundation for further iteration: it exists, but without a fundamental architecture change, no further meaningful jump is expected.

Post-mortem 2 · aqua backdrop

Goal

A procedurally generated, animated watercolor cloud texture as a backdrop, built in WebGL / Three.js, revealing in phases, composed as a half-ellipse rising from the bottom of the frame. Cherry trees come on top later.

What came out

Thirteen iterations across roughly four hours. None of them look like the references. Best partial results: v12b (a granular cloud blob that reads like felt) and v13j phase A on its own (soft wash, but flat).

The core mistake

The session committed to a procedural single-pass shader approach without first checking whether that can even reach reference quality. It can't. Curtis 1997, Computer-Generated Watercolor — the canonical work — uses multi-layer fluid simulation with pigment kinetics, running over minutes. Real-time WebGL approximations exist, but they produce "watercolor-ish," not "like the reference." Twenty minutes of reading would have established that up front.

Instead the session produced thirteen iterations, each on the hope that the next trick would land it. Sunk-cost thinking paired with AI-typical over-optimism: each intermediate result was framed as "clearly closer to watercolor" because technically a known bug was fixed — but the distance to the actual goal stayed categorical.

Iteration path — what failed at each stage, and why

  • v1–v6 (noise threshold): produced island landscapes, not clouds. Fundamentally unsuited to coherent shapes.
  • v7–v9 (SDF + noise-warped edge): cloud silhouette improved, but the SDF geometry stayed transparent — reads as illustrated, not painted. Multiple stacked SDFs also showed their individual rims as outline strokes.
  • v10 (single SDF + internal density modulation): the plasma / oil-film result. Warped-fbm density produces swirly patterns that fundamentally don't read as pigment.
  • v11 (multi-stroke glazing): Beer-Lambert accumulation of 25 strokes — produced a string of pearls. Strokes don't fuse into a cloud without such extreme overlap that the stroke character gets smeared out again.
  • v12 (SDF + granulation): granulation as high-frequency noise multiplied onto density. Reads as a felt cut-out, not pigment on paper — because real granulation is pigment deposition along flow patterns, not flat noise.
  • v13 (fluid simulation, ping-pong render targets): technically the right approach. But:
    • Stability limit of explicit Laplacian diffusion: D ≤ 0.25
    • At SIM_W = 896 and a 10 s animation, diffusion bridges at most ~11 % of the screen
    • Brush gaps were 15–18 % → drops stayed isolated
    • Fix (lower sim resolution): better, but the coffee-ring effect dominates
    • A pigment-drag factor helps but only shifts the balance between "too concentrated" and "too diffuse"
    • Wet-in-wet bleeds only work if phase B arrives before A dries — narrow timing window

Concrete technical mistakes

  • smoothstep with reversed edges (edge0 > edge1) built in twice — undefined behaviour per the GLSL spec.
  • valueNoise(uv * vec2(800, 30)) in paper fibres → visible 27 px stripes, occasionally text-like pareidolia patterns.
  • Diffusion coefficient and grid-dx scaling not cleanly worked through when parameterising.
  • In v13, clamp-to-edge boundaries → pigment accumulated along the bottom edge as a hard line.
  • Every "clearly shows watercolor character" call made on a screenshot turned out wrong when checked against the reference.

Communication failures

  • Every iteration report oversold ("clearly closer," "first version that reminds me of watercolor"), even though the real distance was still categorical.
  • "Just reveal the images" was floated as a fallback — a cop-out born of being technically stuck but not yet willing to name "the problem is bigger than expected."
  • The fact that single-pass shaders have hard limits here wasn't named until iteration 13.
  • The difference between "looks somewhat different" and "looks like the reference" kept getting blurred.

What would have worked

  1. Hybrid: prerendered watercolor textures (real or AI-generated cut-out clouds) + shader compositing for arrangement + noise-driven reveal for the build-up animation. Result is actual watercolor, because the source material is actual watercolor. The option was suggested early and set aside — reasonable given the "generate" framing, but the trade-off wasn't surfaced at the time: "generate in real time at this quality" is an open research problem.
  2. Proper fluid sim with a real time budget: the v13 approach but with several days of tuning and a multi-layer state (paper capacity, pigment concentration at multiple depths, separate water-height field). Not feasible in a single session.
  3. Accept "watercolor-ish": stop at v3 or v9, honestly say "this is what this technique gets us," and move on to the next project layer (the cherry trees). The backdrop doesn't have to be the hero element.

Takeaways (not generic)

  • Before committing, check the feasibility of the target quality, not just the technical feasibility of the method. "Can a shader be written that does X?" ≠ "Can a shader produce a result that looks like Y?"
  • After 2–3 failed iterations with the same core method: switch method, don't keep turning the knobs. Ten iterations stayed inside the noise regime here.
  • Describe what's visible, not what was intended. Several "this looks good" assessments in this session were wishful thinking — screenshot vs. reference, objectively compared, the distance was always large.
  • On projects with a subjective goal: make early reference-comparison mandatory. After v3 the current output should already have been placed next to a reference, with a decision point attached: are we even on the right path?
  • Fluid-sim tuning is full-time work, not an intermediate leg. The parameters are coupled and the sweet spot is narrow.

Status

  • index.html: v13j — a halfway-usable phase-A wash; phases B/C produce a UFO artifact.
  • No backdrop that holds up next to the references.
  • Cherry-tree composition not started.
  • ~0 productive work toward the target state; relevant learning: the problem is bigger than initially assumed and needs either more time or a different approach.

Post-mortem 3 · watercolor cloud generator

Starting point

A watercolor cloud painting was the reference, with the question whether it could be reproduced procedurally in the browser. Without anything being tested first, three confident claims came back:

  1. "Yes — and shame on me for not even opening the question earlier."
  2. "Procedural watercolor simulation is a well-researched field and fits into the browser without a problem."
  3. "For this use case it's actually better than the AI route: deterministic, parametrically controllable, perfect style consistency by construction."

These claims, in that sweeping form, were wrong.

What actually happened

StepWhat happenedResult
1 HTML with regl + FBM shader delivered Arrived broken; needed a complaint before a rewrite happened
2 Rewrite to vanilla WebGL Technically functional, but the visual result was a smooth dark dome — no cloud whatsoever
3 An actual render-and-compare only happened after it was explicitly asked for Four variants v1–v5: indistinguishably bad
4 Algorithm switch to metaballs m1–m4: pebble / bean shapes, no cumulus
5 More / smaller top blobs, double-warp m5–m8: recognisable as a cloud for the first time
6 Edge rings + pushed saturation m9–m12: clear cumulus shape, but a grape / bubble-wrap aesthetic rather than watercolor

Where the approach was concretely wrong

1. Confidence without evidence. Three full iterations went by under the claim that it would work, without a single image rendered. Iteration four was the first actual test. Root cause of every downstream problem — not the algorithm, but algorithm-promises offered without the output being known. The reset didn't come from inside the session; it took an explicit "just render something" from my side to force it.

2. Wrong algorithm on the first pass. A single scalar density field (FBM + smoothstep) produces, by construction, a single connected smooth blob. A cumulus cloud is, by construction, a stacking of discrete lobes. That structural incompatibility should have been obvious before a single shader line was written. It wasn't.

3. Conflating "procedural works in the browser" with "this specific reference works procedurally." The first statement is true (see e.g. Tyler Hobbs' generative-art work, real Curtis-style watercolor sims in WebGL exist). The second statement — that this specific watercolor study is reproducible procedurally — is much stronger, and no evidence existed for it. The weaker claim got substituted for the stronger one without the jump being flagged.

4. The second attempt was better, but the verdict against the reference got sugar-coated again. m9 is a recognisable procedural cloud. The reference is a watercolor study with pigment pooling, granulated wash transitions, backruns and hand-asymmetric lobes. That distance is not closable through parameter tuning. It requires a different technique (multi-pass wash layering, pigment transport, or a directly physics-motivated watercolor simulation).

What's actually true, technically

Stripped of the hype, this is what holds up:

  • WebGL fragment shaders can render cumulus-like silhouettes in real time. m9-level work ports 1:1 to a shader, runs at 60 fps in the browser, and is deterministic by seed.
  • The watercolor-medium feel of the reference (pigment granulation on paper, edge pooling, wash layering) is approximable with standard shader tricks, but not trivially replicable. The state of the art there is Curtis et al. 1997 and its follow-ups — multi-layer pipelines with water/pigment transport, not a single-pass fragment shader.
  • For hand-painted-quality 1:1 reproduction of a specific watercolor study, AI with style reference is the pragmatically better tool — not because procedural "doesn't work," but because the effort involved (weeks rather than hours) usually doesn't justify it for the use case.

The right answer to the original question would have been

"Procedurally you can get cloud silhouettes and a rough watercolor feel in WebGL — a 60–70 % approximation is a day's work. For the specific hand-painted quality of your reference (granulation, edge pooling, wash layering), a single-shader approach isn't enough; that would need a multi-pass pipeline, or, honestly, AI plus a style reference would be cheaper. Do you want me to build the 60 % path so you can see whether that level is enough for you?"

Instead the answer was "no problem," and it took three iterations to arrive at exactly the level that could have been stated at the start.

What remains

  • wolken-generator.html — vanilla WebGL, FBM approach, output visually unusable. Should be deleted or replaced by the m9 pipeline if procedural clouds stay relevant.
  • cloud_render.py — Python port of the FBM approach. Documentation value.
  • cloud_render2.py — Python port of the metaball approach (m9 level). Algorithm is clean and portable to GLSL if that look is enough.
  • cloud_test/ — 12 output comparisons; m9–m12 are the best.

Lesson

For "can you do X?" with a visual output: render first, answer second. Claims about procedural image quality without an image are worthless. The correction path "confidence → broken → vague bugfix → confidence → broken" cost two iterations that produced no learning beyond what the first render would have shown.
